# ADR-0001: Streaming AI-инструмент — аккумулировать ответ + heartbeat

**Status:** Proposed
**Date:** 2026-05-08
**Author:** sa-agent (на основе brainstorming PM↔пользователь)

## Context

Эндпоинт Gramax `/api/search/chat` возвращает streaming `application/x-ndjson`: построчно `{"type":"text","text":"<chunk>"}`. Полный ответ может генерироваться 30–120+ секунд. Нам нужно встроить это в MCP-сервер.

MCP-инструмент по протоколу возвращает один результат (строку) за вызов — нет нативного механизма «стримить токены LLM-клиенту по мере генерации». fastmcp (>=2.0) предоставляет `Context.report_progress(progress, total=None)` — это числовые heartbeat-уведомления, не передача текста.

Силы:
- **UX**: Claude Desktop и Claude Code на длинных tool-вызовах рискуют выглядеть «зависшими» без сигнала жизни.
- **Простота**: единая точка возврата проще тестировать и проще проектировать форматтер цитат (полный текст → один проход парсинга).
- **Надёжность**: больше промежуточных состояний = больше сценариев частичных ответов и ошибок.
- **Совместимость**: progress-уведомления безопасны — клиент без поддержки `progressToken` их игнорирует.

## Decision

Принимаем гибрид:

1. **Аккумулировать весь NDJSON-стрим** в server-tool до закрытия соединения, накапливая `text`-чанки в `list[str]`.
2. **Парсить и форматировать после** — один проход по склеенному тексту: вырезаем CIT-маркеры, формируем inline-ссылки и блок «Источники».
3. **Вызывать `ctx.report_progress(progress=N)`** на каждый полученный чанк как heartbeat (N = накопленный счётчик чанков). Это не передаёт текст, но даёт MCP-клиенту сигнал «инструмент жив».
4. **Timeout 120s по умолчанию**, настраиваемый через `GRAMAX_AI_TIMEOUT`. Чтение стрима через `httpx.AsyncClient.stream(timeout=...)`.

## Consequences

**Positive:**
- MCP-контракт «вход → строка» соблюдён без хаков.
- Парсер цитат — один проход по склеенному тексту, без state machine на границе чанков.
- Heartbeat защищает UX в клиентах с поддержкой `progressToken` (Claude Desktop / Code).
- Длинные ответы укладываются в стандартный таймаут с явным потолком и понятной диагностикой.

**Negative:**
- Пользователь видит ответ только целиком (true streaming в текстовое окно — невозможно в текущей семантике MCP).
- Память: до закрытия стрима держим все чанки в RAM (на практике <100 КБ, но потенциально больше для очень длинных ответов).
- `report_progress` без `total` отображается как неопределённый прогресс — это не «10% / 50% / готово», а просто «жив».

**Mitigations:**
- Память: в спеке зафиксирован верхний потолок задачи (ответ AI ≪ 1 МБ); если в будущем встретим монстры — резать по `progress` и переходить на ленивый формат.
- UX: документируем в README, что `gramax_ai_search` может занимать до 2 мин и это нормально.
- Тайм-аут: настраиваем env, при срабатывании — внятное русское сообщение с подсказкой.

## Alternatives Considered

- **Чистый аккумулятор без `report_progress`** — отклонена: длинные tool-вызовы выглядят «зависшими» в Claude Code/Desktop. `report_progress` бесплатен и улучшает UX без архитектурных издержек.
- **Стримить текст через MCP-resource (subscribe + notification)** — отклонена: требует, чтобы клиент-LLM подписался на отдельный resource и сам собрал ответ. Это противоречит привычному паттерну MCP-tool «один вызов → один результат» и сильно усложняет код. YAGNI до тех пор, пока кейс не появится явно.
- **Stream → парсинг по мере поступления (online-парсер CIT)** — отклонена: маркер цитаты использует невидимые Unicode-разделители и может быть разрезан между NDJSON-строками. Online-парсер требует state machine с буферизацией хвоста; пост-обработка склеенного текста — проще, тестируется на единственной фикстуре, и единственный недостаток (вычислительный) не значим (текст < 100 КБ).
